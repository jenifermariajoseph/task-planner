Developed by: Jenifer Maria Joseph

Technologies: Node.js Express PostgreSQL Docker AWS EC2 Nodemailer

1. Introduction

The Cloud-Ready Daily Planner is a full-stack web application designed to help users manage their daily tasks effectively. It features a custom-designed UI with an interactive animated character, real-time browser notifications, scheduled email reminders, and is fully containerized for cloud deployment.

This project demonstrates end-to-end software development skills — from frontend design and backend API development to database management, containerization with Docker, and production deployment on AWS EC2.

2. Objective

  • Build a production-grade task management application from scratch
  • Implement a RESTful API with CRUD operations backed by PostgreSQL
  • Add real-time reminder systems (browser notifications + email)
  • Containerize the entire stack using Docker and Docker Compose
  • Deploy to a live cloud server (AWS EC2) accessible via public IP

5. Features Implemented

5.1 Task Management (CRUD API)

MethodEndpointDescription
GET/api/tasksFetch all tasks (ordered by latest first)
POST/api/tasksCreate a new task with optional due time
PATCH/api/tasks/:idToggle task completion status
DELETE/api/tasks/:idDelete a task permanently

5.2 Interactive Sun Character

The UI features a cute animated sun character that reacts to task completion:

  • Meh face — when tasks are incomplete or no tasks exist
  • Happy face — when all tasks are marked complete (with a bounce animation)

This gamification element encourages users to complete their tasks, adding a layer of emotional engagement to an otherwise utilitarian tool.

5.3 Browser Notifications

When a task has a due time, the app schedules a browser notification using the Web Notification API. A setTimeout is calculated based on the difference between the due time and the current time, firing an OS-level notification when the task is due.

5.4 Email Reminders (Automated)

A cron job runs every minute on the server, querying for tasks that are:

  • Not completed (completed = false)
  • Past their due time (due_time <= NOW())
  • Not yet emailed (email_sent = false)

To prevent duplicate emails, the system marks email_sent = true before sending the email — an idempotent design pattern that ensures reliability even if the email service is slow.

The frontend uses a warm, hand-drawn aesthetic with:

6. Technology Stack & Justifications

TechnologyPurposeWhy I Chose It
Node.js Server runtime Non-blocking I/O is ideal for handling concurrent API requests and background tasks. JavaScript on both frontend and backend simplifies development.
Express.js Web framework Minimal, flexible, and the most widely-used Node.js framework. Perfect for building REST APIs quickly without unnecessary overhead.
PostgreSQL Database A robust, ACID-compliant relational database. Chose it over MongoDB because tasks have a fixed schema (id, title, due_time, completed, email_sent) that benefits from structured storage and SQL queries.
node-cron Job scheduling Lightweight in-process cron scheduler. No need for external services like Redis or RabbitMQ for a simple per-minute check.
Nodemailer Email delivery The standard Node.js library for sending emails. Gmail App Passwords provide secure SMTP access without exposing the main account password.
Docker Containerization Ensures the application runs identically in development and production. Eliminates "works on my machine" problems by packaging the app with all its dependencies.
Docker Compose Multi-container orchestration Manages both the app and database containers as a single unit with networking, volumes, and environment variables configured declaratively.
AWS EC2 Cloud hosting Industry-standard cloud platform. Provides a full Linux VM with SSH access, giving complete control over the deployment environment.

7. Database Schema

CREATE TABLE tasks (
    id          SERIAL PRIMARY KEY,
    title       TEXT NOT NULL,
    due_time    TIMESTAMP,          -- nullable (tasks without deadlines)
    completed   BOOLEAN DEFAULT false,
    email_sent  BOOLEAN DEFAULT false
);
Design Decision: The due_time column uses TIMESTAMP (not TEXT) to enable proper time comparisons in SQL queries like due_time <= NOW(). The email_sent flag ensures idempotent email delivery.

Timezone Handling: Both containers are configured with TZ: Asia/Kolkata, and PostgreSQL receives an additional -c timezone=Asia/Kolkata flag to ensure NOW() returns IST, matching the user's local timezone for accurate reminder scheduling.

9. AWS EC2 Deployment

  1. Instance Setup: Launched an Ubuntu EC2 instance with a security group allowing inbound traffic on ports 22 (SSH), 3000 (App), and 5432 (PostgreSQL).
  2. SSH Access: Connected via ssh -i planner-key.pem ubuntu@<public-ip>
  3. Docker Installation: Installed Docker and Docker Compose on the EC2 instance.
  4. Code Transfer: Uploaded the project files to the server.
  5. Launch: Ran docker-compose up -d --build to build and start both containers.
  6. Database Init: Created the tasks table inside the running PostgreSQL container.
  7. Result: Application accessible at http://<public-ip>:3000

10. Challenges Faced & Solutions

ChallengeSolution
Timestamp stored as text caused SQL comparison failures Kept TIMESTAMP column type; server converts "HH:MM" input to a proper Date object using setHours() / setMinutes()
Time mismatch between input and display Server builds timestamp using local time methods instead of UTC string parsing. Client reads with new Date(iso).getHours() which auto-converts to local.
Duplicate email reminders Implemented idempotent pattern: set email_sent = true before calling sendMail(), preventing race conditions between cron ticks.
Browser notifications not appearing on macOS Identified that macOS Focus/Do Not Disturb mode was silently blocking notifications. Documented the requirement to disable DND.
Docker container timezone defaulting to UTC Added TZ: Asia/Kolkata environment variable and postgres -c timezone=Asia/Kolkata command to align all services with IST.
Port conflict during Docker deployment Identified and killed the local Node.js process occupying port 3000 before starting Docker containers.
Old docker-compose v1 on EC2 had ContainerConfig bug Worked around by running docker-compose down first, then up --build with fresh containers.

11. Significance of This Project

11.1 Industry-Relevant Skills

This project demonstrates proficiency in the exact tools and workflows used by professional software teams: REST APIs, relational databases, Docker containers, cloud deployment, and automated background processing. Each component was built from scratch, not generated by a framework.

11.2 Production-Grade Practices

  • Environment variable management — credentials never hardcoded; .env excluded from Docker images via .dockerignore
  • Idempotent operations — email reminders cannot be sent twice for the same task
  • Persistent storage — PostgreSQL data survives container restarts via Docker volumes
  • Graceful fallbacks — db.js uses env vars with local defaults, so the app works both locally and in Docker
  • Timezone-aware architecture — all services aligned to the user's timezone for accurate scheduling

11.3 End-to-End Ownership
